# Agent Instructions for `wit`

## Purpose

The `wit` crate contains **auto-generated Rust code from WIT definitions** for the WebAssembly Component Model. It defines the type-safe interface between WASM components (guests) and the Pulumi runtime (host).

**What it does:** Provides generated Rust types and traits for the host-guest WIT interface.

## Architecture Concepts

### WIT Interface Definition
WIT (WebAssembly Interface Types) is a language for describing interfaces between WASM components and their hosts. The `wit/world.wit` file defines:
- What functions the host provides (imports)
- What functions the guest must provide (exports)
- The types used in these functions

### Code Generation
The code is 100% generated by `wit-bindgen` at build time:
- Read `wit/world.wit`
- Generate Rust types (structs, enums, type aliases)
- Generate traits (`Host` for host implementation, `Guest` for guest calls)
- Output to `src/lib.rs` during compilation

### Host and Guest Traits
The generated code provides two main traits:
- **Host trait**: Implement this to provide functions the WASM can call
- **Guest trait**: Use this to call functions in the WASM component

Both are type-safe - the Rust compiler ensures correct usage.

### Opaque Handles
Complex values (like `Output<T>`) can't cross the WASM boundary. Instead, we use `OutputId` (a `u32` handle). The host tracks what each ID refers to, and the guest just passes IDs around.

## Key Concepts

- **WIT definitions**: Interface described in `wit/world.wit`
- **Generated traits**: `Host` and `Guest` for type-safe boundaries
- **Opaque handles**: `OutputId` represents values the guest doesn't own
- **Type conversion**: Automatic between WIT types and Rust types
- **Never edit generated code**: Always modify WIT files instead

## When to Modify

Modify this crate when:
- Adding new operations to the WIT interface
- Changing function signatures
- Adding new types to the WIT definitions
- Updating `wit-bindgen` version

**Critical**: Modify `wit/world.wit`, not `src/lib.rs`!

## Testing Philosophy

The generated code itself doesn't have tests - it's validated by:
- Successful generation (compile-time)
- `wasm_runner` tests (host-side usage)
- WASM example programs (guest-side usage)

If it compiles and the runner works, the bindings are correct.

## Integration Points

- **Generated by `wit-bindgen`**: Code generation tool
- **Used by `wasm_runner`**: Implements the `Host` trait
- **Used by WASM SDKs**: They implement the `Guest` side
- **Defines the contract**: Both host and guest must follow this interface
